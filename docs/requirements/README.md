# Anforderungs-Register & RГјckverfolgbarkeit (Traceability)

> Dieses Verzeichnis ist die Wurzel der **RГјckverfolgbarkeit**. Es ist der Kern
> der Zertifizierungs-FГӨhigkeit (siehe [ADR 0004](../decisions/0004-assurance-und-zertifizierungsfaehigkeit.md)).

## Warum das wichtig ist (fachlich)

Ein Audit nach ED-109A/DO-278A fragt im Kern immer dasselbe: *вҖһZeig mir fГјr jede
Anforderung, wo sie umgesetzt ist вҖ” und wo bewiesen wird, dass sie erfГјllt
ist."* Diese durchgehende Kette nennt man **RГјckverfolgbarkeit**:

```
Anforderung  вҶ’  Design  вҶ’  Code  вҶ’  Test (Nachweis)
        und zurГјck (welcher Test deckt welche Anforderung?)
```

Wer das von Anfang an pflegt, hat spГӨter kein Drama. Wer es nachrГјstet, baut die
Software faktisch neu. Deshalb fangen wir jetzt damit an вҖ” schlank, aber echt.

## Wie wir es konkret machen (technisch)

1. **Jede Anforderung bekommt eine stabile ID** in einer von drei Klassen:
   - `FR-вҖҰ` **Functional Requirement** (was das System *tut*).
   - `NFR-вҖҰ` **Non-Functional Requirement** (Eigenschaften: Cloud, Performance,
     Security, вҖҰ).
   - `CON-вҖҰ` **Constraint** (gesetzte Randbedingung, z. B. Normrahmen).
   IDs werden **nie wiederverwendet**. Wird etwas ungГјltig, Status вҶ’ `verworfen`.

2. **Code und Tests verweisen auf die ID** per Tag-Kommentar, z. B.
   `// REQ: FR-GEO-001` an der umsetzenden Stelle bzw. im Test. So findet man per
   Volltextsuche jederzeit alle BerГјhrungspunkte einer Anforderung.

3. **Die Tabelle unten ist die einzige Wahrheit** (вҖһsingle source of truth").
   Spalte *Nachweis* nennt den/die Test(s), die die Anforderung absichern.

Diese Konvention gilt **ab jetzt** fГјr neuen Code. Bestehender M1-Code wird beim
nГӨchsten Anfassen nachgezogen; seine Nachweise stehen bereits in der Tabelle.
